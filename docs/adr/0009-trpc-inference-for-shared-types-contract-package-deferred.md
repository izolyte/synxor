# tRPC inference as the shared frontend/backend type contract, `packages/contract` deferred

The frontend derives every backend contract type by inferring from the backend `AppRouter`. `apps/api` exposes it through a type-only entrypoint (`src/contract.ts`, wired as the package's `exports` `types`); `apps/web` imports it type-only. `~/lib/trpc.ts` is the single place the frontend touches backend types, exposing `RouterInputs` / `RouterOutputs`; components and hooks infer from there (`RouterOutputs["room"]["create"]`) rather than hand-writing DTOs. This is tRPC's intended model and the documented pattern for a TypeScript monorepo: no codegen, no published package.

`AppRouter` is therefore the single source of truth for every endpoint's input and output. The share resolves from **source, not a built artifact**, so it is live: edit a procedure on the backend and the frontend typecheck — and editor hover — reflect the new shape immediately, with no build step in between. Drift is structurally impossible; a mismatched shape is a red squiggle, not a runtime surprise. It costs no maintenance and scales for free: a new router or procedure appears on the frontend the moment it exists.

Pointing at source has one consequence worth naming. The web `tsconfig` must enable `experimentalDecorators`, because the type-only import pulls the reachable api source — including Nest's `@Injectable()` classes — through the web type-checker. This couples a green web typecheck to api's source compiling cleanly. That is the deliberate trade for live types over a pre-built `.d.ts` snapshot, which would isolate the two but lag behind every backend change until rebuilt. Nothing runtime crosses the boundary: the import is type-only and erases, so the web bundle ships no server code (the package exposes no runtime `exports`, so a stray value import wouldn't even resolve).

A `packages/contract` workspace was considered and **deferred**. It solves a different axis: sharing zod schemas as *runtime values* — reusing a backend input schema to validate a form on the frontend before submit. tRPC inference gives types, not runtime, so it cannot cover that case. But nothing on the frontend needs runtime zod today; creating the package now would be empty ceremony with no consumer.

The two approaches compose rather than compete, and the contract package will also be the clean way to drop `experimentalDecorators` later. When the first form needs shared validation (#8/#9), zod input schemas move out of the routers into a framework-free `packages/contract`; the api imports them via `.input(...)` and the web imports the same schemas for form validation. Defining the router contract framework-free there also lets the web infer `AppRouter` without ever pulling Nest-decorated source, isolating the boundary while staying live. Types still flow through `AppRouter` — the contract package holds schemas (values), never a duplicated set of types. The extraction is mechanical: move the schema file, update the imports. There is no lock-in to writing the schemas inline in the routers first.

Frontend-only types — UI state, internal component props with no backend counterpart — stay local in `apps/web`. The contract package, when it exists, is reserved for what both sides genuinely share.
